# SCHEMA.md Template

This file is generated by the Wiki Agent during `init`. It serves as the
constitution of the wiki — the LLM reads it at the start of every session
to understand how the wiki is structured and what conventions to follow.

## Directory Structure

```
.wiki/
├── SCHEMA.md        — This file. Conventions and workflows.
├── index.md         — Master catalog of all wiki pages.
├── log.md           — Chronological operation log.
├── overview.md      — High-level project synthesis.
├── sources/         — One page per ingested source document.
├── entities/        — Pages for concrete things (modules, APIs, people, tools).
├── concepts/        — Pages for abstract ideas (patterns, principles, themes).
├── analyses/        — Filed query results, comparisons, deep-dive investigations.
└── raw/             — Immutable raw source files (copy or symlink).
    └── assets/      — Images and attachments.
```

## Page Types

### Source Pages (`sources/`)
- One per ingested document
- Named: `YYYY-MM-DD-<slug>.md`
- Contains: summary, key claims, metadata, extracted entities/concepts
- Immutable after creation (don't edit, create corrections in entities/concepts)

### Entity Pages (`entities/`)
- One per concrete entity: a module, API, person, tool, service, component
- Named: `<entity-slug>.md`
- Accumulated from multiple sources
- Updated whenever new information arrives

### Concept Pages (`concepts/`)
- One per abstract concept: a pattern, principle, methodology, theme
- Named: `<concept-slug>.md`
- Synthesized across sources

### Analysis Pages (`analyses/`)
- Filed results from queries, comparisons, investigations
- Named: `YYYY-MM-DD-<slug>.md`
- These are the compound interest of the wiki

## Conventions

### Naming
- Filenames: lowercase, hyphen-separated, no special characters
- Frontmatter `title`: Human-readable, title case
- Tags: lowercase, singular form (e.g., `api` not `APIs`)

### Cross-References
- Use relative markdown links: `[Name](../category/slug.md)`
- Every page should link to at least one other page
- Entity pages link to the sources they were derived from

### Frontmatter Required Fields
```yaml
---
title: Page Title
type: source | entity | concept | analysis | overview | meta
created: YYYY-MM-DD
updated: YYYY-MM-DD
tags: [tag1, tag2]
confidence: high | medium | speculative
status: active | superseded | archived
open_questions: []
contradictions: []
---
```

`confidence` — How trustworthy is the information? `high` = verified from multiple sources, `medium` = single source, `speculative` = inferred.
`status` — Lifecycle state. `superseded` requires linking to the replacement page. `archived` = no longer relevant.
`open_questions` — Unanswered questions this page raises.
`contradictions` — Cross-source conflicts flagged for human review.

### Contradiction Handling
When new information contradicts existing wiki content:
1. DO NOT silently overwrite
2. Add a contradiction notice: `> ⚠️ **Contradiction**: [source-a] claims X, but [source-b] claims Y.`
3. Note which source is more authoritative or recent
4. Flag for human review if both sources seem equally credible

### Update Protocol
1. Read the existing page fully
2. Identify what's new vs. what's already captured
3. Integrate new information into existing structure
4. Add source attribution
5. Update the `updated` date in frontmatter
6. Update `index.md`
7. Append to `log.md`

## Workflows

### Ingest Workflow
1. Read source → discuss with user → create source page
2. Extract entities → create/update entity pages
3. Extract concepts → create/update concept pages
4. Update cross-references, index, overview, log

### Query Workflow
1. Read index → find relevant pages → read them
2. Synthesize answer with citations
3. Offer to file valuable answers as analysis pages

### Lint Workflow
1. Scan for contradictions, orphans, stale pages, gaps, asymmetric links
2. Generate lint report
3. Suggest remediation actions

### Log Format (Parseable)
Every log entry MUST follow this exact format for machine-readability:
```markdown
## [YYYY-MM-DD] <operation> | <description>
- Key detail 1
- Key detail 2
```
Valid operations: `init`, `ingest`, `query`, `lint`, `update`, `archive`

This enables quick inspection with unix tools:
```bash
grep "^## \[" log.md | tail -5          # last 5 operations
grep "ingest" log.md | wc -l            # count ingestions
```

## Adaptation Notes

When generating SCHEMA.md, the agent should:

1. **Fill in project metadata** from auto-detection (name, type, tech stack).

2. **Add domain-specific page types** if appropriate:
   - For code projects: `architecture/` for system design pages
   - For research: `papers/` instead of generic `sources/`
   - For books: `characters/`, `plot-threads/`, `themes/`
   - For business: `decisions/`, `meetings/`, `stakeholders/`

3. **Customize conventions** based on the domain:
   - Code projects: entity pages for modules, classes, APIs
   - Research: entity pages for authors, papers, datasets
   - Personal notes: entity pages for people, goals, habits

4. **Keep it concise.** The schema should fit in one comfortable read. The agent
   will consult it at the start of every session.
